# M2 —— Storyteller（会话级编导 / 压力源）+ Usher + 更强记忆

> 前置阅读:[../architecture.md](../architecture.md)(两平面 / 三层时钟 / 模块蓝图)、
> [../message-ordering.md](../message-ordering.md)(会话/beat/turn/气泡 / 偏序 DAG / 打断两分)。
> 本文把 M2 从老 roadmap 的"响应式 storyteller 在 beat 边界注入压力"**重构**为
> "**storyteller 在会话边界播种意图**"。核心动机:把 storyteller 从**轮询**改成**事件驱动**。

---

## 0. 一句话

**Storyteller 不再 mid-conversation 插手。** 它在一段 conversation **自然结束**时被唤醒一次,
为下一段 conversation 播种**意图**(闲聊 / 推进某剧情 / 化解张力…),交给 conductor 执行。
僵局 → 会话自然收束 → storyteller 自然介入。用户的爆炸性输入 → usher 判 `user_forced` →
提前收束当前会话 → 走同一条唤醒路径。

## 1. 为什么这样改(架构级动机)

老 M2 让 storyteller"响应式、在节拍边界介入"——那本质要求它**盯着每一拍**(轮询),
重模型高频空转。改到**会话边界触发**后:

- **轮询 → 事件驱动**:唤醒频率从"每拍"降到"每会话"。这才是"storyteller 主动打断开销较重"的真正根源
  (不只是 token,是 polling vs event-driven 的架构差)。
- **叙事更顺**:一段对话有形状(起→展开→僵/高潮→收)。边界落子尊重形状,mid-conversation 硬插把形状拽歪。
- **边界延迟是特性**:真实群聊换话题前本就有停顿;重模型思考下一段意图时的 lull,读起来像自然冷场。

## 2. 三层嵌套时钟(各一个 owner)

```
Storyteller  —— 会话级时钟（低频、重）：给一段 conversation 定意图，边界唤醒
Conductor    —— beat 级时钟（高频、轻）：拿意图，在会话内排 beat、检测会话该不该结束
Delivery     —— 气泡级时钟（微观）：typing / 停顿 / 交错
```

"conductor 从 storyteller 收到 conversation 意图"就是最上面那层交接。

## 3. 会话状态机(请求 / 应答)

```
IDLE ──► storyteller 播种 ConversationIntent ──► conductor 跑会话（若干 beat）
   ▲                                                    │
   └──────── storyteller 读 ConversationEnd 再播种 ◄─────┘   （conductor 检测到结束 + 原因）
```

会话是一个**跨会话的循环**:seed → run → end(带原因) → reseed。storyteller 只在 seed/reseed 两点工作。

### 数据契约(草案,放 `domain/`)

```python
@dataclass
class ConversationIntent:      # storyteller → conductor
    kind: str                  # chitchat | develop_plot | resolve_tension | ...
    hook: str = ""             # 这段空气里悬着什么（注入 agent 尾部的"压力指令"）
    focus: list[str] = field(default_factory=list)   # 可选：点名相关角色
    length_budget: int | None = None                 # 粗略预期 beat 数（尤其闲聊，无自然高潮）
    tension_target: float | None = None

@dataclass
class ConversationEnd:         # conductor → storyteller
    reason: str                # lull | deadlock | intent_fulfilled | user_forced | max_length
    tension: float             # 结束时张力读数
    summary_hook: str = ""     # 这段发生了什么，供 storyteller 记忆 + 播种下一段
```

## 4. 会话结束检测(M2 唯一要啃的新启发式)

conductor 不能只说"结束了",**必须报 `reason`**——原因决定 storyteller 下一手:

| reason | 含义 | storyteller 下一手 |
| --- | --- | --- |
| **lull** | 闲聊耗尽,没人有新话 | 抛新钩子 / 换话题 |
| **deadlock** | 张力卡死、原地打转(**僵局**) | 抛压力 / 转折 / 炸弹**打破** |
| **intent_fulfilled** | 意图达成 | 收束,进下一段 |
| **user_forced** | 用户关键输入强制收束 | 把用户输入并进新意图(含世界抗拒) |
| **max_length** | 超长硬兜底 | 强制收,防拖死 |

`lull` 与 `deadlock` 是两种相反的结束,却都唤醒 storyteller——这就是"僵局时 storyteller 自然介入"的落点。

**初版检测先土**:conductor 连续 N 拍返回 None = `lull`;张力高但 M 拍无进展 = `deadlock`;
`max_beats` 硬兜底。先粗后精,别一开始追求聪明。检测器过急会把对话剁碎、过松会拖沓,是最该调的旋钮。

## 5. Usher —— 用户输入的台口分流

用户消息进来,由 **usher**(戏剧班底命名:前厅引位,门槛分流)判一次:

- **判据 = "这句需不需要世界回应?"**,**不是**"激不激进"。canon 冲突 = 世界必须回应(哪怕它很安静)。
  用这个轴,安静的设定破坏也会被捕获,而不是只抓响亮的炸弹。
- **absorb(附和)** → 不升级:进共享历史,当前会话继续,下个边界 storyteller 自然看到。
- **escalate** → `user_forced`:提前收束当前会话,唤醒 storyteller 播种回应用户的新意图。

**关键性质:误判只赔延迟,不赔丢失。** 被 absorb 的输入照样进历史,下个边界一定被 storyteller 看到。
所以 usher 可以**调保守**——拿不准就放行,边界兜底,不必怕漏。

放置:`message/` 内、贴着 conductor(产出既喂 conductor 的会话内路由,又喂 storyteller 的边界升级)。

> 老 M2 的"用户输入意图分类(推进/捣乱/试探/拐弯)"折进 usher,作为 escalate 时附带的方向标签。

## 6. 世界连贯性:抗拒下沉到边界

老 M2 让 storyteller 用"世界抗拒"mid-conversation 否决破坏设定的输入。现在:

- **用户造成的连贯性问题 ⟺ `user_forced`**(§5 判据保证)。所以"世界抗拒"变成 **storyteller 在边界的行为**:
  用户扔破坏设定的炸弹 → `user_forced` → 会话收束 → storyteller 播种的下一段让世界抵抗/困惑/不弯折。
  和 `user_forced` 走同一条路,**不需要单独的每拍 canon 守卫**(用户路径)。
- **硬边界(保留老 M2)**:storyteller **只控制世界,不替用户做决定**;张力读数——低张力推一把、高张力后撤,让张力呼吸。

## 7. 打断的归属(收紧 message-ordering 的结论)

- **storyteller 现在完全不做打断**——只跨会话播种。
- 会话内的**剧本化打断/交错**(作者气泡 DAG 的边)由 **conductor** 执行意图时顺手编(廉价 barge-in 当质感)。
- **用户 `user_forced`** 是唯一能提前终结会话的反应式事件,退化成"提前触发一次正常的边界交接",机制统一。

## 8. 更强记忆 + 会话持久化

- storyteller 要"记得"过去几段(意图是什么、怎么结束的)才能播种连贯的下一段。
- store 加一张 **`conversations` 表**(id, room_id, intent, end_reason, tension, summary, ts),消息挂 `conversation_id`。
  这是 roadmap SQLite 模型的一个小增量。
- 记忆增强(从老 M2):比 M1 的"JSON 行追加"更结构化的私有记忆合并/去重(具体策略 M2 中细化,不阻塞骨架)。

## 9. 明确不做(本期预留)

- **AI 自己说崩的 canon 漂移**:没有 `user_forced`,usher 管不着。**独立小轨,本期不建**;将来三选一——
  便宜的每拍连贯性 check / 留到下个边界让 storyteller 自纠 / 靠 prompt 纪律压住。先记着别忘。
- 重 storyteller 的**多步 agentic 推理 / tool / sim 数值**:属暗线内部,不进消息流(两平面不变式)。M2 只要它出 Intent。
- 跨窗回复 RAG、成本定价:延续既有"预留不建"。

## 10. 落地分期(骨架先行,重模型后接)

现在代码里**没有"会话"概念**——只有扁平消息流 + 每拍 conductor。所以:

- **A｜会话骨架**:runtime/conductor 引入**会话状态机 + 结束检测器**(先 lull/max_beats 规则,`reason` 先分 lull/deadlock/forced)。
  `ConversationIntent` 给固定"闲聊"stub,走通 seed→run→end→reseed。意图经现有 `conductor_instruction` 尾部槽注入 prompt。
- **B｜Usher**:用户输入分流(absorb / `user_forced`)。判据"世界要不要回应",先规则/便宜模型,可保守。
- **C｜Storyteller(重模型)**:把 stub 意图换成真 storyteller;张力读数 + 世界抗拒(边界播种)。
- **D｜会话持久化 + 记忆增强**:`conversations` 表;私有记忆结构化合并。

每期落地即绿。A 能在**完全不碰重模型**的前提下把"会话边界"这个新结构立起来测通,风险最小。

## 11. 验证

- **A**:离线 mock 跑一段——观察 conductor 在 N 拍无新意后以 `reason=lull` 收束、reseed 起新会话;`max_beats` 兜底生效。
- **B**:构造"安静的 canon 破坏"输入,断言 usher 判 `escalate/user_forced`(验证判据是"世界要不要回应"而非"激进度");
  普通附和输入判 `absorb`、会话继续。
- **C**:僵局场景(两角色对峙 M 拍无进展)→ conductor 报 `deadlock` → storyteller 播种一段打破僵局的意图。
- **缓存不变式**:意图注入走尾部不缓存层,共享历史前缀仍逐字节稳定;复核 `test_prompt_builder` + warm-cache 断言。
- **两平面**:确认 storyteller 的 agentic/tool 都在暗线内部,**不进前台消息流**。
